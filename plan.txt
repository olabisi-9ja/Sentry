=== Concept ===
Sentry
A Community Intelligence Brief Powered by Gemma 4
Consolidated Concept Report — Build with Gemma: GenAI for SDGs (KWASU)
1. Executive Summary
Sentry is a WhatsApp-native community intelligence system for KWASU. Students report issues — safety incidents, power/water outages, transport, sanitation — the same way they already do: by texting. Gemma 4 classifies, clusters, triages, and summarizes these reports in real time, with no new app to download and no account to create. Urgent reports are escalated instantly to responders; routine reports become a shared, browsable community feed. No model training is involved — Gemma 4 works entirely through prompting and retrieval over live report data (RAG).
2. Problem Statement
Campus issues are reported late, duplicated, or lost across scattered WhatsApp groups and word-of-mouth. Security staff have no aggregated view of patterns or hotspots. Students have no single place to check what's actually happening around them. Existing tools solve one narrow problem each and never become reusable infrastructure.

4. How Gemma 4 Powers It (No Training Required)
Every capability below is achieved by sending Gemma 4 a prompt with the relevant report(s) as context. The model's weights never change; each request is independent.
Task
How Gemma 4 is used
Classification
Report text/transcript → category + severity, via a single structured prompt.
Clustering / dedup
Recent reports fed into the prompt; Gemma identifies which describe the same incident.
Summarization
A batch of reports → concise brief (daily/hourly), grounded only in supplied data.
Ask mode (RAG)
Student question + retrieved recent reports → direct, sourced answer.
Translation / local language
Handles mixed Pidgin/English input and responds in kind.
Emergency triage
Urgency detection at intake; flags routed for immediate human escalation.
5. Core Features
5.1 Report Intake — WhatsApp Front End
Students text a known WhatsApp number (Twilio) — zero download, zero new trust ask.
Anonymous by default; no account or name required.
Supports text, voice notes (transcribed), and photos.
5.2 Emergency Triage & Escalation
Gemma 4 classifies urgency at intake — high-urgency reports skip the normal queue.
Urgent reports are instantly routed to a human responder (security dispatch / on-call staff) — Gemma never dispatches help autonomously.
Every routed alert includes Gemma's classification + original text for accountability.
5.3 Community Broadcast / Digest
Gemma 4 generates a daily/hourly brief from the day's reports.
Digest pushed back to a WhatsApp broadcast list — closes the loop from individual reports to shared awareness.
5.4 Ask Mode
Students can text a question (e.g. "is there water in Hall B?") and get a direct, sourced answer via RAG over recent reports.
5.5 Public News / Browse Feed
Verified, non-sensitive reports become readable feed items — Gemma 4 rewrites messy input into clean blurbs.
Filter by category, location, recency, and status (ongoing/resolved).
Access via WhatsApp command ("news", "updates") or a simple public web page — no login required.
Privacy: identifying details (names, numbers) are stripped by Gemma 4 before anything is made public.

7. Scaling Vision
Widen categories: sanitation, transport, lost items, events — same pipeline, larger taxonomy.
Role-based views: students (WhatsApp only), admins (dashboard), service providers (filtered feed).
Multi-community growth: swap community_id — each campus/estate/NGO gets its own number and Gemma instance over its own data.
This is the original Cortex thesis: one brain, reusable across many communities and use cases.
8. Privacy & Safety Boundaries
No autonomous emergency dispatch — Gemma classifies and routes, a human always confirms and acts.
Anonymization applied before any public-facing summary or feed item.
Every derived output (classification, cluster, brief) is reproducible from the raw report and stated model version.
9. MVP Demo Scope (1-Day Build)
Twilio WhatsApp webhook → FastAPI → Gemma 4 classification, live.
One seeded batch of sample reports → Gemma-generated daily brief.
One urgent-report path → visible escalation routing.
One public feed view (web page or WhatsApp "news" command) showing cleaned, filtered reports.
Judges can text the live number and watch classification + brief generation happen in real time.
10. Why This Comes Out on Top
Meets students where they already are (WhatsApp) — no adoption barrier, no new trust ask.
Uses Gemma 4 exactly as the hackathon expects: prompting + RAG, no training, no GPU cluster.
Directly demoable end-to-end in real time, not just a static writeup claim.
Clear, defensible SDG 11 / SDG 16 story — safety and shared community awareness for an underserved setting.
Scoped to one day, with an honest, credible scaling roadmap for judges who reward vision beyond the demo.

=== Plan ===
Sentry
Full Build Plan: From Campus Pilot to Community Platform
Phased roadmap — hackathon MVP through full local-community deployment
1. Scope Clarification
Sentry is not a campus-only tool. KWASU is the pilot — the first place the system runs — but the actual goal is a reusable intelligence layer for the wider community around it (Malete, Ilorin, and eventually other towns and institutions), following the same community_id architecture already defined in the Sentry concept report. This document lays out how to get from the July 25 hackathon build to that full platform without losing either target along the way.
2. Guiding Principle
Two timelines have to be held at once: an 11-day hackathon deadline, and a multi-month platform build. The rule that keeps both alive: everything shipped for July 25 must also be the real first brick of the full platform — not a throwaway demo. Nothing in Phase 0 gets rebuilt from scratch later; it gets extended.

4. Phase 0 — Hackathon MVP (Now → July 25)
This phase does not change based on the wider vision — it stays deliberately narrow so it actually ships. Full detail is in the existing Sentry Concept Report and Kaggle writeup; summarized here for continuity:
WhatsApp intake (Twilio) → FastAPI → Gemma 4 classification, clustering, confidence scoring, summarization
One campus, one channel, one reasoning pipeline — no Telegram, no app, no heatmap math yet
Seeded demo data + a live number judges can text
Kaggle writeup, public repo, and live demo submitted July 25
5. Phase 1 — Harden the KWASU Pilot (Aug–Sep 2026)
Turn the demo into something people actually rely on daily, still scoped to campus.
Move off a sandbox Twilio number onto a persistent WhatsApp Business number
Replace the mock confidence/verification logic with a real lightweight human review queue — recruit 2–3 students or staff as first verifiers
Basic authentication on the admin dashboard (this was designed in Phase 0's UI already — wire it to real accounts)
Add logging/monitoring so failures are visible, not silent
Success metric: real reports flowing weekly, and a rough sense of classification accuracy from verifier feedback
6. Phase 2 — Local Community Expansion (Oct–Dec 2026)
This is where the project stops being campus-only. The hardest part of this phase is trust — a university has a captive, shared-identity audience; a town does not.
6.1 New Intake Sources
Local government contact point (LGA liaison) as an official channel, not just informal reports
Community leaders, market associations, and transport unions as trusted local verifiers — this replaces "security staff" as the primary verification layer outside campus
6.2 Category Expansion
Add categories relevant off-campus: road conditions, local utility escalation, community security patrols
6.3 Language
Extend Gemma 4 prompts to handle local languages spoken across Kwara State, not just English/Pidgin — test this explicitly before rollout, don't assume it works
6.4 Trust-Building
Publish the confidence-scoring methodology publicly — people trust systems they can inspect
Publicize a clear moderation/privacy policy before asking non-students to report sensitive things
7. Phase 3 — Multi-Community Platform (2027+)
This is the original Cortex thesis: one brain, reusable everywhere, monetized as infrastructure.
Multi-tenancy via community_id — onboard additional campuses, estates, or towns without re-architecting
Self-serve or assisted onboarding flow for a new community's admin team
Move storage from a single Postgres instance to something built for multi-tenant scale; add caching to control Gemma 4 call costs as volume grows
Commercial model: direct licensing to campuses/estates/local governments, plus API access for partner developers — as outlined in the existing Cortex Business Model document
8. Team & Resourcing
Phase 0–1 is buildable solo or with a small team. Phase 2 introduces work that isn't engineering at all — community outreach, verifier recruitment, moderation policy — and will bottleneck a solo builder faster than the code will. Identify before Phase 2 starts: who handles partnerships and verifier relationships, since that can't be automated the way classification can.
